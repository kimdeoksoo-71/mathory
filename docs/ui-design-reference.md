# Mathory UI 디자인 레퍼런스 가이드

> **목적**: UI 전면 개편 시 참고할 디자인 원칙과 레퍼런스를 정리한 문서  
> **작성일**: 2026-03-22  
> **적용 시기**: 에디터 기능 안정화 이후 (목표: 2026년 5~6월)  
> **파일 위치**: `docs/ui-design-reference.md`

---

## 1. 디자인 4원칙

Mathory의 모든 UI 판단은 아래 4가지 기준으로 우선순위를 매긴다.

| 순위 | 원칙 | 설명 |
|------|------|------|
| 1 | **본문이 주인공** | 문제·해설 텍스트와 수식이 화면의 중심. 나머지 요소는 숨기거나 축소 |
| 2 | **차분한 신뢰감** | 튀는 색상·애니메이션 배제. 뉴트럴 톤, 일관된 간격, 안정적 타이포그래피 |
| 3 | **모던 심플** | 불필요한 장식 제거. 여백·타이포그래피·색상만으로 위계를 표현 |
| 4 | **통일성** | 색상 팔레트, 서체, 간격, 라운딩, 아이콘 스타일이 앱 전체에서 일관 |

---

## 2. 레퍼런스 앱 10선

### 2.1 Typora — typora.io

**핵심 키워드**: 콘텐츠가 곧 인터페이스, WYSIWYM

**Mathory에 적용할 아이디어**:

- **에디터 영역 제로 크롬**: 편집 모드에서 툴바·메뉴를 최대한 숨기고, 본문 텍스트와 수식만 보이게 한다. 현재 MathToolbar는 항상 노출되는데, 수식 블록에 커서가 들어갈 때만 플로팅으로 나타나는 방식을 검토.
- **타이포그래피 리듬**: 제목(heading) → 본문 → 수식 → 보기(list) 간 font-size·line-height·margin 비율을 체계적으로 정의. Typora의 기본 테마(GitHub, Newsprint)를 측정해서 비율 추출.
- **포커스 모드**: 현재 편집 중인 블록만 100% 불투명, 나머지 블록은 50~60% 투명도로 처리. iA Writer의 문장 단위 포커스를 블록 단위로 변형.
- **마크다운 문법 기호 처리**: `#`, `*`, `-` 등 문법 기호를 회색으로 살짝 보여주되 시각적 방해를 최소화. 비활성 줄에서는 문법 기호를 숨기는 Typora 방식 참고.

---

### 2.2 Linear — linear.app

**핵심 키워드**: 뉴트럴 톤, 일관된 크롬, 차분한 밀도

**Mathory에 적용할 아이디어**:

- **사이드바 디자인**: 뉴트럴 그레이 배경, 아이콘+텍스트 조합, 접이식 토글. 현재 Mathory 사이드바의 폴더 트리·문제 목록을 Linear의 레이아웃 패턴으로 재설계.
- **헤더 바 구성**: 최상단에 얇은 헤더 바. 왼쪽에 브레드크럼(폴더 > 문제명), 오른쪽에 액션 버튼(저장, 내보내기, 설정) 최소한으로 배치.
- **컨텍스트 메뉴**: 우클릭/3점 메뉴의 스타일을 Linear처럼 통일. 둥근 모서리, 그림자, 호버 하이라이트 일관성.
- **다크/라이트 모드**: Linear의 2024~2025 리디자인에서 적용한 "따뜻한 그레이(warm neutral)" 팔레트 참고. 순수 블랙(#000) 대신 다크 그레이(#1a1a1a~#222), 순수 화이트(#fff) 대신 오프화이트(#fafafa) 사용.
- **색상 시스템**: 모노크롬 기반에 액센트 컬러 1~2개만 사용. 액센트는 링크, 선택 상태, 주요 CTA에만 제한적으로 적용.

---

### 2.3 Notion — notion.so

**핵심 키워드**: 블록 에디터, 트리 네비게이션, / 커맨드

**Mathory에 적용할 아이디어**:

- **빈 상태(Empty State) 디자인**: 새 문제를 생성했을 때 빈 화면에 "문제를 작성해보세요" 같은 안내 텍스트 + 단축키 힌트를 보여주는 방식. 현재는 빈 에디터만 보이는데, 온보딩 요소를 추가.
- **블록 핸들**: 각 블록(텍스트, 수식, 이미지) 왼쪽에 드래그 핸들(⋮⋮)을 호버 시에만 표시. 블록 재정렬을 직관적으로.
- **/ 커맨드 팔레트**: 현재 MathToolbar 드롭다운 외에, `/수식`, `/이미지`, `/표` 같은 슬래시 커맨드로 블록 삽입을 지원하는 방안 검토.
- **폴더↔페이지 전환 애니메이션**: 사이드바에서 폴더 선택 시 부드러운 전환. 급격한 화면 교체 대신 fade 또는 slide 적용.

**주의**: Notion의 기능 과잉은 피한다. "Notion의 구조 + Typora의 절제"가 목표.

---

### 2.4 Craft — craft.do

**핵심 키워드**: 카드형 블록, 부드러운 그림자, Apple 감성

**Mathory에 적용할 아이디어**:

- **문제 카드 디자인**: 문제 목록에서 각 문제를 카드로 표시할 때, Craft의 카드 스타일 참고. 얇은 보더 또는 미세한 그림자, 충분한 내부 패딩, 호버 시 살짝 들어올리는 효과.
- **블록 간 시각적 구분**: 문제 본문 안에서 텍스트 블록, 수식 블록, 이미지 블록이 자연스럽게 구분되되 경계선 없이 여백과 배경색 차이만으로 분리.
- **라이트/다크 모드 전환**: Craft의 부드러운 모드 전환 애니메이션 참고. 색상이 급변하지 않고 0.2~0.3초간 서서히 전환.
- **문서 내 멀티미디어 조화**: 이미지, 텍스트, 수식이 한 문서 안에서 시각적으로 균형을 이루는 방식. 이미지 블록의 라운딩, 캡션 스타일 등.

---

### 2.5 iA Writer — ia.net/writer

**핵심 키워드**: 극단적 미니멀, 타이포그래피 집착, 포커스 모드

**Mathory에 적용할 아이디어**:

- **서체 전략**: iA Writer는 자체 서체(Mono/Duo/Quattro)를 개발할 정도로 타이포그래피에 투자. Mathory도 에디터 영역의 기본 서체를 신중하게 선택해야 한다. 한글 본문용(Pretendard, Noto Sans KR 등)과 수식용(KaTeX 기본) 서체의 조화를 테스트.
- **포커스 모드 심화**: iA Writer의 "문장 포커스" 개념을 Mathory에 맞게 변형. 수식 편집 중에는 해당 수식 블록만 하이라이트되고 주변 텍스트는 희미해지는 방식.
- **에디터 전체 화면 모드**: 사이드바·헤더를 모두 숨기고 에디터만 전체 화면으로 확장하는 "Zen Mode". 문제 작성에 몰입할 때 사용.
- **커서·캐럿 디자인**: iA Writer의 두꺼운 블록 커서가 시각적 앵커 역할을 하듯, Mathory에서도 수식 편집 커서의 가시성을 높이는 방법 검토.

---

### 2.6 Overleaf — overleaf.com

**핵심 키워드**: 소스+미리보기 2패널, LaTeX 네이티브, 파일 트리

**Mathory에 적용할 아이디어**:

- **2패널 구조 옵션**: 현재 Mathory는 소스/미리보기가 블록 단위로 전환되지만, 고급 사용자를 위해 Overleaf처럼 좌우 분할(소스 | 미리보기) 모드를 옵션으로 제공하는 방안.
- **수식 오류 표시**: Overleaf의 컴파일 에러 패널처럼, Mathory의 LaTeX 린팅 결과를 에디터 하단 또는 사이드 패널에 정리해서 보여주는 방식.
- **파일 트리 사이드바**: Overleaf의 왼쪽 파일 트리는 Mathory의 폴더 구조와 유사. 다만 비주얼은 Linear/Notion 스타일로.

**주의**: Overleaf의 비주얼 스타일 자체는 레거시 느낌이 강하므로, 구조와 기능 배치만 참고하고 외관은 다른 레퍼런스를 따른다.

---

### 2.7 Stripe Docs — docs.stripe.com

**핵심 키워드**: 기술 문서의 황금 표준, 여백·위계·가독성

**Mathory에 적용할 아이디어**:

- **콘텐츠 유형별 시각 분리**: 코드 블록은 어두운 배경, 설명 텍스트는 밝은 배경, 경고/팁은 색상 바로 구분. Mathory에서 "문제 본문 / 수식 블록 / 이미지 / 메타 정보"를 각각 다른 시각 처리.
- **타이포그래피 위계**: Stripe Docs의 heading 크기·굵기·색상·간격 체계를 Mathory의 문제 구조(대문항 → 소문항 → 보기)에 매핑.
- **사이드 네비게이션**: 좌측에 전체 목차, 우측에 현재 페이지 내 소목차(TOC). Mathory에서 긴 문제/해설 문서를 편집할 때, 우측에 블록 단위 미니맵을 표시하는 방안.
- **반응형 여백**: 화면 폭에 따라 본문 영역의 max-width와 좌우 패딩이 자연스럽게 조절되는 방식. 본문이 너무 넓어지지 않도록 650~750px 정도로 제한.

---

### 2.8 Bear — bear.app

**핵심 키워드**: 따뜻한 절제, 3단 구조, 태그 시스템

**Mathory에 적용할 아이디어**:

- **3단 레이아웃**: 사이드바(폴더) → 문제 목록(리스트) → 에디터(본문). Bear의 3단 구조가 Mathory의 정보 흐름과 가장 유사. 각 단의 너비 비율, 구분선 스타일, 접기/펼치기 동작 참고.
- **따뜻한 컬러 톤**: Bear의 빨간 액센트 + 크림색 배경 조합은 차분하면서도 개성 있다. Mathory도 뉴트럴 그레이에 따뜻한 액센트 1개(예: 차분한 블루 또는 틸)를 선택.
- **마크다운 문법 기호 투명화**: Bear가 편집 모드에서 `#`, `**` 등을 회색으로 처리하는 방식. Mathory에서도 `$`, `\frac{}{}` 같은 LaTeX 문법 기호를 에디터 내에서 시각적으로 약화.
- **태그 → 메타 정보 매핑**: Bear의 해시태그 시스템을 Mathory의 메타 정보(대단원, 난이도)에 대입. 문제에 `#미적분 #4점` 같은 태그를 달고, 사이드바에서 태그별 필터링.

---

### 2.9 Dropbox Paper — paper.dropbox.com

**핵심 키워드**: 빈 종이 느낌, 플로팅 서식 도구, 콘텐츠 퍼스트

**Mathory에 적용할 아이디어**:

- **플로팅 툴바**: 텍스트를 선택했을 때만 서식 툴바가 선택 영역 위에 플로팅으로 나타나는 방식. MathToolbar를 이 패턴으로 전환하면 평소에는 에디터 영역이 완전히 깨끗해진다.
- **최소 UI 상태**: 문서를 열면 제목 + 빈 본문만 보이고, 사이드바도 기본 접힌 상태. 사용자가 필요할 때 펼치는 구조.
- **빈 문서 안내**: "이름을 지어주세요" (제목), "여기에 작성하세요" (본문) 같은 플레이스홀더 텍스트로 빈 문서에 생동감 부여.
- **미디어 삽입 UX**: 이미지를 드래그 앤 드롭으로 넣으면 자동 업로드·삽입되는 흐름. Mathory의 이미지 블록 삽입도 이처럼 매끄럽게.

---

### 2.10 GitBook — gitbook.com

**핵심 키워드**: 문서 관리 플랫폼, 좌측 목차+우측 TOC, 검색

**Mathory에 적용할 아이디어**:

- **좌측 목차 + 우측 페이지 TOC**: GitBook의 듀얼 네비게이션. 좌측은 전체 문서(폴더) 구조, 우측은 현재 문서 내 섹션 이동. Mathory에서 긴 해설을 편집할 때 유용.
- **검색 UX**: Cmd+K로 열리는 커맨드 팔레트 형태의 검색. 문제 제목, 본문 내용, 태그를 통합 검색. 현재 Mathory의 찾기/바꾸기를 이 방향으로 확장.
- **버전 관리 UI**: GitBook의 변경 이력 비교 화면. Mathory에서 문제 수정 이력을 보여주는 기능을 추후 구현할 때 참고.
- **다크/라이트 모드**: GitBook의 자연스러운 모드 전환과 중간 톤 처리 참고.

---

## 3. 종합: 영역별 레퍼런스 매핑

각 UI 영역을 설계할 때 **1순위로 참고할 앱**을 정리한다.

| UI 영역 | 1순위 레퍼런스 | 2순위 | 핵심 포인트 |
|---------|--------------|-------|------------|
| **레이아웃 골격** (사이드바 + 메인) | Linear | Bear | 접이식 사이드바, 헤더 바, 2~3단 구조 |
| **에디터 영역** (본문 편집) | Typora | iA Writer | 크롬 최소화, 본문=인터페이스, 포커스 모드 |
| **수식 편집 UX** | Overleaf | — | 2패널 옵션, 오류 표시, 수식 툴바 배치 |
| **블록/카드 디자인** | Craft | Notion | 카드 스타일, 블록 구분, 드래그 핸들 |
| **서식 도구 표시 방식** | Dropbox Paper | Notion | 플로팅 툴바, 숨김 기본, 필요 시 노출 |
| **컬러 시스템** | Linear | Bear | 뉴트럴 그레이 기반, 따뜻한 액센트 최소 사용 |
| **타이포그래피** | iA Writer | Stripe Docs | 서체 선택, 크기·간격 체계, 가독성 |
| **콘텐츠 유형 시각 분리** | Stripe Docs | — | 본문/수식/이미지/메타 각각 다른 처리 |
| **네비게이션/검색** | GitBook | Notion | 목차 트리, Cmd+K 검색, 페이지 내 TOC |
| **빈 상태/온보딩** | Dropbox Paper | Notion | 플레이스홀더, 안내 텍스트, 단축키 힌트 |

---

## 4. 컬러 시스템 초안

Linear + Bear를 기반으로 한 초안. 실제 적용 시 조정 필요.

```
[Light Mode]
배경(메인)       #fafafa
배경(사이드바)    #f5f5f5
배경(수식 블록)   #f0f0f0
텍스트(본문)     #1a1a1a
텍스트(보조)     #6b6b6b
텍스트(비활성)    #b0b0b0
보더             #e5e5e5
액센트(주)       #3b82f6  (차분한 블루)
액센트(위험)     #ef4444  (삭제, 오류)
액센트(성공)     #22c55e  (저장 완료)
```

## 6. 간격(Spacing) 체계

8px 그리드 기반으로 통일한다.

```
xs:   4px   (아이콘 내부 패딩)
sm:   8px   (인접 요소 간 최소 간격)
md:  16px   (블록 내부 패딩, 리스트 항목 간)
lg:  24px   (섹션 간 간격)
xl:  32px   (영역 간 대구분)
2xl: 48px   (페이지 상하 여백)
```

## 7. 실행 순서 (UI 작업 시)

에디터 기능 안정화 이후, 아래 순서로 UI 작업을 진행한다.

**Phase A: 기반 작업**
1. 컬러 토큰을 CSS 변수로 정의 (라이트/다크)
2. 타이포그래피 스케일 정의
3. 간격 시스템 정의 (8px 그리드)
4. 기본 컴포넌트 스타일 통일 (버튼, 인풋, 드롭다운, 모달)

**Phase B: 레이아웃**
5. 사이드바 리디자인 (Linear 참고)
6. 헤더 바 리디자인
7. 에디터 영역 여백·max-width 조정 (Stripe Docs 참고)
8. 3단 레이아웃 반응형 처리 (Bear 참고)

**Phase C: 에디터 UX**
9. MathToolbar → 플로팅 전환 (Dropbox Paper 참고)
10. 블록 핸들·블록 구분 디자인 (Craft/Notion 참고)
11. 포커스 모드 구현 (Typora/iA Writer 참고)
12. 빈 상태·플레이스홀더 디자인

**Phase D: 확장**
13. 다크 모드 구현
14. Cmd+K 커맨드 팔레트 (GitBook 참고)
15. Zen Mode (전체 화면 편집)
16. 마이크로 애니메이션·전환 효과
