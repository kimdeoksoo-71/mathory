## ADR-001: 수식 렌더링 — KaTeX 선택

**날짜**: 2026-02-16
**상태**: 확정

### 배경
수학 문제 플랫폼에서 LaTeX 수식을 실시간 렌더링해야 함.

### 후보
- **KaTeX**: 빠른 렌더링 속도, 서버/클라이언트 모두 사용 가능
- **MathJax**: 더 넓은 LaTeX 커버리지, 느린 렌더링

### 결정
KaTeX 선택.

### 이유
- 실시간 미리보기에서 체감될 정도의 속도 차이 (KaTeX가 약 10배 빠름)
- CSAT 수학 범위의 수식은 KaTeX로 충분히 커버 가능
- 추후 MathJax에서만 지원하는 기능이 필요하면 재평가

## ADR-002: 에디터 — CodeMirror 6 선택

**날짜**: 2026-02-16
**상태**: 확정

### 후보
- **CodeMirror 6**: 모듈화, 가볍고, 커스텀 확장 용이
- **Monaco (VS Code)**: 강력하지만 번들 크기가 큼
- **Textarea + 라이브러리**: 단순하지만 확장성 부족

### 결정
CodeMirror 6 선택.

### 이유
- snippet 기능으로 Tab placeholder 구현 가능 (수식 템플릿 핵심)
- Markdown 언어 지원 내장
- 번들 크기가 Monaco 대비 훨씬 작음
- 커스텀 구문 하이라이팅 확장 가능 (LaTeX)

## ADR-003: 데이터 저장 — raw_text만 저장

**날짜**: 2026-02-16
**상태**: 확정

### 결정
Firestore에 Markdown + LaTeX 원문(raw_text)만 저장. 렌더된 HTML은 저장하지 않음.

### 이유
- 렌더링 라이브러리 변경 시 마이그레이션 불필요
- DB 저장 용량 절약
- 원문이 있으면 언제든 다시 렌더링 가능 (SSR/CSR 무관)
- 검색 시 원문 기반 텍스트 검색이 더 정확

### 트레이드오프
- 페이지 로드마다 클라이언트에서 렌더링 필요 (KaTeX 속도로 충분)

## ADR-004: 블록 저장 — subcollection vs embedded array

**날짜**: 2026-02-16
**상태**: 확정 (재평가 가능)

### 결정
question_blocks, solution_blocks를 subcollection으로 분리.

### 이유
- 개별 블록 수정 시 전체 문서 재작성 불필요
- 블록 수가 늘어나도 Firestore 문서 크기 제한(1MB)에 자유로움
- 블록별 권한 제어 가능성

### 트레이드오프
- 문제 전체 로드 시 3회 쿼리 필요 (문서 + 2개 subcollection)
- 실사용에서 블록 수가 적고(~10개) 항상 전체를 읽는 패턴이면 embedded array가 더 단순할 수 있음

### 재평가 조건
- Phase 2에서 실제 CRUD 패턴을 보고 embedded로 전환 여부 결정
